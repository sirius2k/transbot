# Claude 작업 시간 측정 전략

## 개요

Claude와 함께 작업할 때 시간 측정은 여러 복잡한 요소를 고려해야 합니다. 이 문서는 WORKLOG 시간 측정을 위한 실용적인 전략을 제시합니다.

## 1. 시간 측정의 핵심 원칙

### 권장 접근법: Task 기반 벽시계 시간 (Task-based Wall Clock Time)

```text
작업 시작 시점 → [Claude 응답 대기 + 검토 + 승인 + 재작업] → 작업 완료 시점
└─────────────────── 모두 작업 시간에 포함 ──────────────────┘
```

### 선택 이유

- Claude 응답 대기 시간도 작업의 일부
- 검토/승인도 필수 작업 과정
- 측정이 단순하고 일관성 유지 가능
- 실제 프로젝트 타임라인과 일치

## 2. 시간 측정 기준점

### 2.1 시작 시점 (Start Time)

다음 중 하나가 발생한 시점:

1. **Claude에게 첫 요청을 보낸 시점**

```text
예: "app.py에 복사 버튼을 추가해줘" → 2026-02-03 11:05 기록
```

2. **작업 계획을 시작한 시점** (복잡한 작업의 경우)

```text
예: FEATURE-017 구현 계획 수립 시작 → 2026-02-03 10:00 기록
```

### 2.2 종료 시점 (End Time)

다음이 **모두** 완료된 시점:

1. Claude의 작업 완료 확인
2. 사용자의 검토 및 승인 완료
3. 필요한 경우 테스트 실행 완료
4. Git 커밋 완료 (코드 작업인 경우)

```text
예: 커밋 완료 → 2026-02-03 11:25 기록
```

## 3. 특수 상황 처리 규칙

### 3.1 중단 및 휴식

**규칙**: 10분 이상 작업을 중단한 경우 별도 기록

```markdown
| CODE | 복사 버튼 구현 | 2026-02-03 11:05 | 2026-02-03 11:25 | 20m |
| - | [휴식] | 2026-02-03 11:25 | 2026-02-03 11:40 | - |
| CODE | 복사 버튼 스타일 수정 | 2026-02-03 11:40 | 2026-02-03 11:50 | 10m |
```

**합계 계산**: 휴식 시간 제외 (20m + 10m = 30m)

### 3.2 Claude 재시도/반복 작업

**규칙**: 처음 요청부터 만족스러운 결과까지 모두 포함

```markdown
| CODE | API 에러 핸들링 구현 | 2026-02-03 14:00 | 2026-02-03 14:30 | 30m |
# 첫 시도 (10분) + 검토 후 수정 요청 (15분) + 재검토 (5분) = 30분
```

### 3.3 병렬 작업

**규칙**: 각 작업을 개별 라인으로 기록

```markdown
| DOC | README 수정 | 2026-02-03 10:00 | 2026-02-03 10:15 | 15m |
| CODE | 버그 수정 | 2026-02-03 10:15 | 2026-02-03 10:30 | 15m |
# 두 작업 간 컨텍스트 스위칭 포함
```

### 3.4 분석/리서치 작업

**규칙**: Claude와의 대화형 탐색도 시간에 포함

```markdown
| ANALYSIS | Azure OpenAI 설계 분석 | 2026-02-03 09:00 | 2026-02-03 10:30 | 1h 30m |
# Claude에게 질문 + 응답 검토 + 추가 질문 + 문서 읽기 모두 포함
```

## 4. 작업 분류별 시간 측정 가이드

| 작업분류 | 시작 시점 | 종료 시점 | 포함 사항 |
|---------|----------|----------|----------|
| **DOC** | 문서 작성/수정 요청 | 문서 완성 및 검토 완료 | Claude 응답 대기, 검토, 수정 요청 |
| **CODE** | 코드 작성/수정 요청 | 코드 완성, 테스트 통과, 커밋 | Claude 응답 대기, 검토, 재작업, 커밋 |
| **TEST** | 테스트 작성/실행 시작 | 모든 테스트 통과 확인 | 테스트 작성, 실행, 디버깅, 수정 |
| **ANALYSIS** | 분석/조사 시작 | 결론 도출 및 문서화 완료 | Claude 대화, 코드 읽기, 문서 읽기 |
| **ETC** | 기타 작업 시작 | 작업 완료 | 설정, 브랜치 생성 등 |

## 5. 실용적인 기록 방법

### 5.1 실시간 기록 템플릿 (권장)

작업 중 메모장에 실시간 기록:

```text
[2026-01-31 11:05] CODE START: 복사 버튼 구현
[2026-01-31 11:25] CODE END: 복사 버튼 구현 완료 및 커밋
[2026-01-31 11:30] TEST START: 복사 버튼 테스트 작성
[2026-01-31 11:40] TEST END: 테스트 통과
```

이후 WORKLOG 파일로 정리:

```markdown
| CODE | 복사 버튼 구현 | 2026-01-31 11:05 | 2026-01-31 11:25 | 20m |
| TEST | 복사 버튼 테스트 작성 | 2026-01-31 11:30 | 2026-01-31 11:40 | 10m |
```

### 5.2 Git 커밋 메시지 활용

커밋 시 주석으로 시간 기록:

```bash
git commit -m "feat: 복사 버튼 추가

<!-- Task 시작: 11:05, 종료: 11:25, 소요: 20분 -->
```

## 6. 예측 시간 vs 실제 시간 비교

각 Task에 예측 시간 추가:

```markdown
| 작업분류 | 작업내용 | 예측 | 시작 | 종료 | 실제 | 차이 |
| -------- | --------- | ------ | ------ | ------ | ------ | ------ |
| CODE | 버튼 구현 | 30m | 2026-02-03 11:05 | 2026-02-03 11:25 | 20m | -10m ✅ |
| TEST | 테스트 작성 | 15m | 2026-02-03 11:30 | 2026-02-03 11:50 | 20m | +5m ⚠️ |
```

## 7. WORKLOG 템플릿 개선 제안

기존 템플릿에 예측 시간 컬럼 추가:

```markdown
# FEATURE-XXX 작업 시간 로그

| 작업분류 | 작업내용 | 예측 | 시작 시간 | 종료 시간 | 실제 | 차이 |
| -------- | --------- | ------ | --------- | --------- | ------ | ------ |
| DOC | 계획 수립 | 20m | 2026-02-03 10:00 | 2026-02-03 10:15 | 15m | -5m |
| CODE | 구현 | 1h | 2026-02-03 10:15 | 2026-02-03 11:30 | 1h 15m | +15m |
| TEST | 테스트 | 30m | 2026-02-03 11:30 | 2026-02-03 11:45 | 15m | -15m |

## 소요시간 합계

- DOC: 15분 (예측 20분, -5분)
- CODE: 1시간 15분 (예측 1시간, +15분)
- TEST: 15분 (예측 30분, -15분)
- **총 소요시간**: 1시간 45분 (예측 1시간 50분, -5분)

## 인사이트

- CODE 작업이 예측보다 15분 더 소요됨 (재시도 2회)
- TEST 작업은 예측보다 빨리 완료 (기존 테스트 재사용)
```

## 8. 권장 워크플로우

```text
1. Task 시작 → 예측 시간 기록 (선택)
   ↓
2. 시작 시간 기록 (HH:MM:SS)
   ↓
3. Claude에게 요청 → 응답 대기 → 검토 → 승인/수정
   ↓
4. 작업 완료 → 종료 시간 기록
   ↓
5. 실제 소요 시간 계산
   ↓
6. 예측 vs 실제 비교 분석 (선택)
```

## 9. 시간 측정의 목적

### 주요 목적

1. **예측 정확도 향상**: 반복 작업을 통해 예측 시간을 더 정확하게 산정
2. **병목 지점 파악**: 어떤 작업 유형이 예측보다 오래 걸리는지 분석
3. **프로젝트 계획 수립**: 향후 유사 프로젝트의 일정 계획에 활용
4. **생산성 개선**: 시간이 오래 걸리는 작업의 원인 분석 및 개선

### 측정하지 말아야 할 것

- Claude 응답 속도 자체 (제어 불가)
- 순수 사고 시간 (측정 불가)
- 학습 시간 (별도 분류 권장)

## 11. 주의사항

### 시간 측정 시 피해야 할 것

1. **과도한 정밀도**: 초 단위까지 기록할 필요 없음 (분 단위로 충분)
2. **실시간 업데이트 강박**: 작업 완료 후 일괄 기록도 괜찮음
3. **완벽주의**: 추정치이므로 ±5분 오차는 허용
4. **시간 압박**: 측정이 목적이 아닌 수단임을 기억

### 효과적인 시간 측정을 위한 팁

1. **일관성**: 동일한 기준으로 측정
2. **단순성**: 복잡한 규칙보다 간단한 규칙 선호
3. **유연성**: 상황에 따라 규칙 조정 가능
4. **회고**: 주기적으로 시간 로그 리뷰 및 개선

## 12. 로그 파일 관리 방식

### 12.1 단일 파일 vs 다중 파일 비교

#### 단일 파일 관리 방식

**장점**:

- **단순성**: 모든 로그가 한 곳에 있어 관리가 쉬움
- **시간순 조회**: 전체 프로젝트의 시간 흐름을 한눈에 파악
- **검색 용이성**: 하나의 파일에서 모든 내용 검색 가능 (Ctrl+F)
- **전체 통계 산출**: 한 파일에서 전체 작업 시간 집계 용이
- **파일 관리 부담 감소**: 여러 파일을 열고 닫을 필요 없음
- **히스토리 추적**: 시간 순서대로 작업 흐름 파악 용이

**단점**:

- **파일 크기 증가**: 프로젝트가 커지면 파일이 매우 커질 수 있음
- **FEATURE별 필터링**: 특정 FEATURE만 보려면 검색 필요
- **로딩 속도**: 파일이 커지면 에디터에서 열 때 느려질 수 있음

#### 다중 파일 관리 방식

**장점**:

- **명확한 구분**: FEATURE별 파일 분리로 관리가 명확
- **1:1 매칭**: FEATURE-XXX.md ↔ WORKLOG-FEATURE-XXX.log
- **파일 크기 관리**: 각 파일이 적정 크기 유지
- **아카이빙 용이**: 완료된 FEATURE는 로그도 함께 보관

**단점**:

- **파일 관리 복잡**: 여러 파일을 생성하고 관리해야 함
- **전체 통계**: 여러 파일에서 데이터를 모아야 함
- **시간순 조회**: 전체 타임라인 파악이 어려움

### 12.2 권장: 단일 파일 통합 관리

TransBot 프로젝트 특성상 **단일 파일 방식**을 권장합니다.

**선택 이유**:

1. **개인 프로젝트 규모**: 파일 크기가 크게 문제되지 않음
2. **전체 타임라인 중요**: 시간순 작업 흐름 파악이 더 중요
3. **Markdown 섹션 활용**: 섹션으로 FEATURE별 그룹핑 가능
4. **검색 효율성**: Ctrl+F로 충분히 필터링 가능
5. **관리 단순화**: 하나의 파일만 업데이트하면 됨

### 12.4 작업 구분 체계 (2차원 분류)

#### 1차원: 작업 범위

- **FEATURE-XXX** (F-XXX): 2시간 이상 소요되거나 여러 파일 수정이 필요한 주요 기능
- **Quick Win** (QW-XX): 1-2시간 이내 완료 가능한 간단한 개선사항

#### 2차원: 작업 방식

- **직접**: 커맨드라인에서 Claude에게 직접 명령 (일반적인 대화)
- **Skill**: `.claude/commands/` 폴더의 스킬 사용 (예: commit-and-push)

#### ID 체계

- FEATURE: `F-009` (FEATURE-009의 약어)
- Quick Win: `QW-01` (날짜별로 01부터 시작)
