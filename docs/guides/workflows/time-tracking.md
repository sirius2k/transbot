# 작업 시간 추적 가이드

TransBot 프로젝트의 작업 시간 추적 방법을 안내합니다.

## 작업 분류 체계

모든 작업은 명확한 유형으로 분류하여 시간을 추적합니다. ISSUE 로그와 TASK 로그 모두 동일한 분류 체계를 사용합니다.

### 기본 4가지 분류

| 분류 | 태그 | 설명 | 예시 |
| ---- | ---- | ---- | ---- |
| **문서작업** | `[DOC]` | 문서 읽기, 작성, 업데이트, 계획 수립 | README 수정, 설계 문서 작성, FEATURE 계획 수립 |
| **코딩작업** | `[CODE]` | 코드 작성, 수정, 리팩토링 | 함수 구현, 클래스 수정, 버그 수정 |
| **테스트작업** | `[TEST]` | 테스트 작성, 실행, 디버깅, 검증 | 단위 테스트, 커버리지 확인, 린트 실행 |
| **기타** | `[ETC]` | 위 3가지에 속하지 않는 작업 | 브랜치 생성, PR 생성, 환경 설정 |

### 선택적 5번째 분류

| 분류 | 태그 | 설명 | 예시 |
| ---- | ---- | ---- | ---- |
| **분석작업** | `[ANALYSIS]` | 코드베이스 리서치, 설계 분석 | 아키텍처 조사, 성능 프로파일링 |

**참고**: 간단한 분석은 `[DOC]`에 포함 가능 (예: "문서 분석 및 계획 수립")

## 로그 파일 명명 규칙

모든 작업 로그는 **TASK 패턴**으로 통일하여 관리합니다.

### TASK 패턴 (통합 로그 형식)

- **파일명**: `FEATURE-XXX-TASK-X.Y.log`
- **예시**: `FEATURE-008-TASK-8.5.log`
- **사용 시점**: 모든 작업 (GitHub Issue 유무 무관)
- **특징**: GitHub Issue는 로그 내부 필드로 선택적 명시

## GitHub Issue 연동

### 작업에 GitHub Issue가 있는 경우

```text
## 작업 정보
- Task 번호: 8.5
- Task 제목: setup_sidebar() 함수 모델/Deployment 목록 분리
- GitHub Issue: #14       ← Issue 번호 명시
- 작업 브랜치: issue-14
- PR 번호: #21
```

### 작업에 GitHub Issue가 없는 경우

```text
## 작업 정보
- Task 번호: 8.8
- Task 제목: 문서 업데이트
- GitHub Issue: 없음      ← "없음" 명시
- 작업 브랜치: main
- PR 번호: 없음
```

## 로그 작성 예시

### 예시 1: GitHub Issue가 있는 경우

```text
# FEATURE-009-TASK-9.2 작업 시간 로그

## 작업 정보
- Task 번호: 9.2
- Task 제목: 번역 히스토리 저장 기능 구현
- GitHub Issue: #15
- 작업 브랜치: issue-15
- PR 번호: #23
- 작업 날짜: 2026-02-01

## 작업 시간 상세

### [ANALYSIS] 코드베이스 분석
- 시작: 2026-02-01T10:00:00+09:00
- 완료: 2026-02-01T10:15:00+09:00
- 소요: 15분

### [DOC] 설계 문서 작성
- 시작: 2026-02-01T10:15:00+09:00
- 완료: 2026-02-01T10:30:00+09:00
- 소요: 15분

### [CODE] HistoryManager 클래스 구현
- 시작: 2026-02-01T10:30:00+09:00
- 완료: 2026-02-01T11:00:00+09:00
- 소요: 30분

### [TEST] 단위 테스트 작성
- 시작: 2026-02-01T11:00:00+09:00
- 완료: 2026-02-01T11:20:00+09:00
- 소요: 20분

### [ETC] 환경 변수 설정
- 시작: 2026-02-01T11:20:00+09:00
- 완료: 2026-02-01T11:25:00+09:00
- 소요: 5분

## 작업 시간 요약

### 유형별 합산
- 분석작업 (ANALYSIS): 15분
- 문서작업 (DOC): 15분
- 코딩작업 (CODE): 30분
- 테스트작업 (TEST): 20분
- 기타 (ETC): 5분

### 총 작업 시간
- **실제 소요 시간**: 85분 (약 1시간 25분)
- **순수 작업 시간**: 80분 (분석+문서+코딩+테스트)
```

- 완료: 2026-02-01T14:15:00+09:00
- 소요: 15분

### [DOC] CLAUDE.md 업데이트

- 소요: 15분

## 작업 시간 요약

### 유형별 합산

### 총 작업 시간
- **실제 소요 시간**: 30분
- **순수 작업 시간**: 30분 (문서 작업만)

## 분류 선택 가이드

### DOC vs ANALYSIS 구분

#### [DOC] 사용: 문서 읽기/작성이 주목적

- "README.md 업데이트"
- "FEATURE 계획 수립"
- "문서 분석 및 계획 수립" (분석이 간단한 경우)

#### [ANALYSIS] 사용: 깊은 조사/분석이 주목적

- "아키텍처 패턴 조사"
- "성능 병목 지점 분석"
- "의존성 트리 분석"
- "이슈 원인 심층 분석"

### CODE vs TEST 구분

#### [CODE] 사용: 프로덕션 코드 작성

- 함수/클래스 구현
- 버그 수정
- 리팩토링

#### [TEST] 사용: 테스트 관련 모든 작업

- 단위 테스트 작성
- 테스트 실행
- 커버리지 확인
- 린트/타입 체크
