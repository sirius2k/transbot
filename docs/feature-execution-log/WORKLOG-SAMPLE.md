# WORKLOG 작성 규칙

## 개요

TransBot 프로젝트의 모든 작업 시간을 단일 파일(`WORKLOG.md`)에 기록하여 관리합니다.

- **파일 위치**: `docs/feature-execution-log/WORKLOG.md`
- **관리 방식**: 단일 파일 통합 관리
- **기록 원칙**: Task 기반 벽시계 시간 (Claude 응답 대기, 검토, 재작업 모두 포함)

## 작업 분류 체계 (2차원)

모든 작업은 **2차원 분류 체계**로 기록합니다.

### 1차원: 작업 범위

- **FEATURE-XXX** (ID: F-XXX): 2시간 이상 소요되거나 여러 파일 수정이 필요한 주요 기능
- **Quick Win** (ID: QW-XX): 1-2시간 이내 완료 가능한 간단한 개선사항

### 2차원: 작업 방식

- **직접**: 커맨드라인에서 Claude에게 직접 명령 (일반적인 대화)
- **Skill**: `.claude/commands/` 폴더의 스킬 사용 (예: commit-and-push)

### 작업 분류 (5가지)

1. **문서작업** (DOC): 문서 읽기, 작성, 업데이트, 계획 수립
   - 예시: README 수정, 설계 문서 작성, FEATURE 계획 수립
2. **코딩작업** (CODE): 코드 작성, 수정, 리팩토링
   - 예시: 함수 구현, 클래스 수정, 버그 수정, 코드 개선
3. **테스트작업** (TEST): 테스트 작성, 실행, 디버깅, 검증
   - 예시: 단위 테스트, 통합 테스트, 커버리지 확인, 린트 실행
4. **분석작업** (ANALYSIS): 코드베이스 리서치, 설계 분석, 조사
   - 예시: 아키텍처 조사, 의존성 분석, 성능 프로파일링, 이슈 원인 분석
5. **기타** (ETC): 위 4가지에 속하지 않는 모든 작업
   - 예시: 브랜치 생성, PR 생성, 환경 설정, 의존성 설치, 커밋 및 푸시

## 로그 파일 템플릿

### FEATURE 작업 로그 예시

```markdown
## FEATURE-009: 환경 변수 기반 설정 관리 시스템

### 작업 내역

| ID | 작업분류 | 작업방식 | 작업내용 | 예측 | 시작 | 종료 | 실제 | 차이 |
|----|---------|----------|---------|------|------|------|------|------|
| F-009 | CODE | 직접 | 설정 모듈 설계 및 구현 | 1.5h | 20:00 | 21:30 | 1.5h | 0m ✅ |
| F-009 | DOC | 직접 | .env.example 업데이트 | 30m | 21:30 | 22:00 | 30m | 0m ✅ |
| F-009 | CODE | 직접 | TranslationManager 적용 | 1h | 20:44 | 20:46 | 2m | -58m ✅ |
| F-009 | CODE | 직접 | app.py에 설정 적용 | 1h | 20:49 | 00:00 | 11m | -49m ✅ |
| F-009 | TEST | 직접 | 단위 테스트 작성 | 1.5h | 00:14 | 00:18 | 4m | -1h 26m ✅ |
| F-009 | DOC | Skill | 문서 업데이트 및 커밋 | 1h | 00:19 | 00:25 | 6m | -54m ✅ |

### 소요시간 합계

- CODE: 2시간 7분 (예측 4시간 15분, -2시간 8분)
- DOC: 36분 (예측 1시간 30분, -54분)
- TEST: 4분 (예측 1시간 30분, -1시간 26분)
- **총 소요시간**: 2시간 27분 (예측 8시간, -5시간 33분)

### 인사이트

- Claude 활용으로 예측 대비 5시간 33분 절감 (69% 단축)
- 특히 테스트 작성에서 1시간 26분 절감 (예측 1.5h → 실제 4m)
- Skill 사용으로 문서 업데이트 및 커밋 작업 효율화 (6분 소요)
```

## 컬럼 설명

### 필수 컬럼

| 컬럼명 | 설명 | 예시 |
|--------|------|------|
| **ID** | 작업 식별자 | F-009 (FEATURE-009), QW-01 (Quick Win #1) |
| **작업분류** | 5가지 분류 중 하나 | DOC, CODE, TEST, ANALYSIS, ETC |
| **작업방식** | 직접 또는 Skill | 직접, Skill |
| **작업내용** | 작업 설명 | "Config 클래스 구현" |
| **예측** | 예상 소요 시간 | 1.5h, 30m, 10m |
| **시작** | 시작 시간 (HH:MM) | 20:00, 09:45 |
| **종료** | 종료 시간 (HH:MM) | 21:30, 10:15 |
| **실제** | 실제 소요 시간 | 1.5h, 25m |
| **차이** | 예측 대비 차이 | -30m ✅, +10m ⚠️ |

### 시간 표기 방법

- **시각**: `HH:MM` 형식 (예: 20:00, 09:45)
- **시간**: `Xh Ym` 또는 `Xm` 형식 (예: 1h 30m, 45m, 2h)

### 차이 표시

- **예측보다 빠름**: `-Xm ✅` 또는 `-Xh Ym ✅`
- **예측과 동일**: `0m ✅`
- **예측보다 느림**: `+Xm ⚠️` 또는 `+Xh Ym ⚠️`

## 워크플로우

### 1. 작업 시작

1. `docs/feature-execution-log/WORKLOG.md` 파일 열기
2. 해당 섹션 찾기:
   - FEATURE 작업: 해당 `## FEATURE-XXX` 섹션
   - Quick Win: `## Quick Wins > ### YYYY-MM-DD` 섹션

### 2. 작업 기록

**실시간 기록 권장**:

```text
[2026-01-31 11:05] F-009 CODE START: Config 클래스 구현
[2026-01-31 12:35] F-009 CODE END: Config 클래스 구현 완료
```

### 3. 작업 완료 후

1. 테이블에 로그 추가
2. 소요시간 합계 업데이트
3. 인사이트 작성 (선택)
4. 전체 통계 섹션 업데이트 (주기적)

## 전체 통계 섹션

`WORKLOG.md` 파일 최상단에 위치:

```markdown
## 📊 전체 통계 요약

- 총 작업 시간: 7시간 36분
- FEATURE 완료: 2개 (FEATURE-009, FEATURE-008)
- Quick Wins 완료: 3개
- 마지막 업데이트: 2026-02-01 15:30
```

## ID 체계

### FEATURE ID

- **형식**: `F-XXX` (FEATURE-XXX의 약어)
- **예시**: F-009 (FEATURE-009), F-010 (FEATURE-010)

### Quick Win ID

- **형식**: `QW-XX` (날짜별로 01부터 시작)
- **예시**: QW-01, QW-02, QW-03
- **리셋**: 매달 또는 일정 개수 이후 01부터 다시 시작

## 주의사항

### 시간 측정 원칙

1. **벽시계 시간 측정**: Claude 응답 대기, 검토, 재작업 모두 포함
2. **시작 시점**: Claude에게 첫 요청 또는 작업 계획 시작 시점
3. **종료 시점**: 작업 완료, 검토, 테스트, 커밋 모두 완료 시점
4. **10분 이상 중단**: 별도 로그 라인으로 기록 (작업분류: `-`, 작업내용: `[휴식]`)

### 작업 기록 팁

1. **일관성**: 동일한 기준으로 측정
2. **단순성**: 분 단위 측정으로 충분 (초 단위 불필요)
3. **유연성**: ±5분 오차는 허용
4. **실시간 기록**: 작업 완료 직후 기록 (메모장 활용 권장)
